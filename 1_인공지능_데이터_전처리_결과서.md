# 📄 산출물 1: 인공지능 데이터 전처리 결과서

## 1. 프로젝트 개요

### 1-1. 비즈니스 배경

YouTube 트렌딩 섹션은 알고리즘이 선별한 상위 200개 영상이 실시간으로 노출되는 공간이다. 이 때, 트렌딩 지속 시간은 카테고리에 따라 수 시간에서 수십 일까지 수십 배 차이가 난다. <br>단순히 인기 있는 영상을 찾는 것이 아니라, **인기가 얼마나 오래 지속되는지**가 광고 효율의 핵심 변수다.

본 프로젝트는 "트렌딩 차트에 얼마나 오래 남는가"를 데이터로 예측함으로써, 카테고리별 집행 타이밍과 대상 선택의 효율을 극대화하는 근거를 제시하는 것을 목표로 한다.

### 1-2. 비즈니스 목표

YouTube KR 트렌딩 데이터(2022~2025)에서 카테고리 특성과 트렌딩 진입 초반 반응 신호를 기반으로 트렌딩 지속성을 예측하는 모델을 구축하고, 카테고리간 차이가 통계적으로 유의한지 검증한다. 
<br>궁극적으로 "어떤 카테고리 채널에, 언제, 얼마나 집행해야 노출 대비 효율이 극대화되는가"를 데이터로 답한다.

### 1-3. 연구 질문 (RQ)

| RQ | 질문 | 분석 방법 |
|----|------|----------|
| RQ1 | 카테고리 그룹 간 트렌딩 지속 시간은 통계적으로 유의미한 차이가 있는가 | `category_group → trending_duration_h` 차이 검정 (Kruskal-Wallis) |
| RQ2 | 트렌딩 초반 반응이 지속 시간을 예측할 수 있는가 | T0-only, 24h ML, DL 회귀·분류 모델 |
| RQ3 | 트렌딩이 기준 이상 지속될지 분류할 수 있는가 | T0-only, 24h MLP 분류 모델, tdi_label |
| RQ4 | 같은 카테고리 안에도 서로 다른 트렌딩 패턴이 존재하는가 | K-Means 클러스터링 + 사후 EDA |

### 1-4. 머신러닝 활용 계획

| 항목 | 내용 |
|------|------|
| 문제 유형 | 회귀 + 이진 분류 |
| 핵심 예측 대상 | `tdi_label` (지속성·순위 품질 복합 성과 라벨, 이진 분류) |
| 보조 예측 대상 | `trending_duration_h` (회귀), `long_label_cat_q75` (이진 분류 비교용) |
| 모델 구조 | T0-only 모델 vs T0+24h 모델 — 독립적으로 학습 후 성능 비교 |
| 알고리즘 | Ensemble (XGBoost, Catboost, LightGBM) |
| 해석 방법 | SHAP — 피처별 기여도 정량화 |
| 활용 방안 | 카테고리별 광고 집행 최적 타이밍 및 기간 권고 |

### 1-5. 성공 기준 (Success Criteria)



| 모델 | 지표 | 목표값 |
|------|------|--------|
| T0-only 회귀 | MAE | 12시간 이하 |
| T0-only 분류 (tdi_label) | AUC-ROC | 0.75 이상 |
| T0+24h 분류 (tdi_label) | AUC-ROC | T0-only 대비 개선 |

**모델 분리 설계의 핵심 의미:** T0-only와 T0+24h 모델의 성능 차이가 곧 "진입 후 24시간 조회수 성장 정보(`view_growth_24h_log`)가 예측력을 얼마나 향상시키는가"에 대한 정량적 근거가 된다.

## 2. 데이터셋 소개

### 2-1. 데이터 기본 정보

| 항목 | 내용 |
|------|------|
| 데이터셋명 | Global YouTube Trending Dataset (2022–2025) |
| 출처 | Illinois Data Bank: IDB-9307654 · Ng & Goncalves (2025) · arXiv:2510.23645 · CC BY 4.0 |
| 수집 기간 | 2022년 7월 ~ 2025년 6월 |
| 수집 방식 | YouTube Data API 기반 하루 4회 스냅샷 수집 (약 6시간 간격) |
| 필터링 조건 | `region_code = KR` (한국 트렌딩만 사용) |
| 원본 스냅샷 수 | 약 872,191개 행 |
| 분석 단위 | 트렌딩 이벤트 (`video_id + event_id` 복합키), 약 32,317개 |
| 저장 포맷 | Parquet |

**스냅샷과 이벤트의 구분:**
원본 데이터는 동일 영상이 트렌딩에 머무는 동안 6시간마다 행이 추가되는 스냅샷 구조 
<br>같은 영상이 3일 동안 트렌딩에 있으면 약 12개 행이 생성 
<br>분석의 실제 단위는 이 스냅샷이 아니라 진입~이탈 전 과정을 1개 행으로 집계한 **트렌딩 이벤트**.

### 2-2. 원천 컬럼 목록 및 설명

| 컬럼명 | 데이터 타입 | 설명 | 사용 여부 |
|--------|------------|------|----------|
| `collection_date` | DATETIME | 스냅샷 수집 시각 (하루 4회) | ✅ 핵심 (UTC→KST 변환) |
| `region_code` | VARCHAR | 지역 코드 (KR 고정) | 상수 |
| `rank` | SMALLINT | 트렌딩 순위 1~200 | ✅ 핵심 피처 원천 |
| `video_id` | VARCHAR | 영상 고유 ID | ✅ PK |
| `title` | TEXT | 영상 제목 | ✅ 보조 피처 원천 |
| `published_at` | DATETIME | 업로드 일시 | ✅ 핵심 피처 원천 |
| `channel_id` | VARCHAR | 채널 고유 ID | ✅ 사용 |
| `channel_title` | VARCHAR | 채널명 | ✅ 사용 |
| `category_id` | SMALLINT | YouTube 카테고리 번호 | ✅ 핵심 (5그룹 재분류) |
| `default_language` | VARCHAR | 영상 언어 (null 多) | ❌ 제거 |
| `view_count` | BIGINT | 스냅샷 시점 누적 조회수 | ✅ 핵심 피처 원천 |
| `comment_count` | INT | 스냅샷 시점 누적 댓글수 | ✅ 사용 (like_count 부재로 유일한 engagement 원천) |
| `description` | TEXT | 영상 설명 | ❌ 제거 |
| `default_audio_language` | VARCHAR | 오디오 언어 | ❌ 제거 |
| `live_broadcast_content` | VARCHAR | 라이브 여부 | ❌ 제거 |

**없는 컬럼 주의:**
- `like_count`: YouTube가 2021년 이후 API에서 비공개 처리. 데이터에 존재하지 않음. engagement 지표는 `T0_comment / T0_view`로 대체.


### 2-3. KR 트렌딩 데이터의 구조적 특성

`region_code = KR` 트렌딩에는 한국어 콘텐츠 외에도 외국 채널 영상이 상당수 포함. 
<br>Tottenham Hotspur(Sports), Jimmy Kimmel Live(Entertainment), Charlie Puth(Music) 등 
<br>영어권 채널의 영상이 실제로 KR 트렌딩에 등장하며, `default_language`가 'en'이거나 `<NA>`인 경우가 대다수.

- KR 트렌딩 판정은 YouTube 알고리즘이 한국 사용자의 시청 패턴을 기반으로 결정. 외국 채널 영상이라도 KR 트렌딩에 포함된 것은 실제 한국 사용자들의 시청 결과.

### 2-4. 타겟 변수 분포 (전처리 후 기준)

| 타겟 변수 | 유형 | 분포 요약 |
|----------|------|----------|
| `trending_duration_h` | 연속형 (회귀) | 중간값 약 114h, 평균 약 146h, 최솟값 6h, 최댓값 768h, 오른쪽 긴꼬리 분포 |
| `tdi_label` | 이진 분류 (메인) | TDI ≥ 0.4 → 1 (약 35%), TDI < 0.4 → 0 (약 65%) |

## 3. 탐색적 데이터 분석 (EDA)

### 3-1. 기초 통계량

**trending_duration_h 요약**

| 통계량 | 값 |
|--------|-----|
| 전체 이벤트 수 | 약 32,317개 |
| 최솟값 | 6시간 (단일 스냅샷 이벤트 하한 보정값) |
| 25분위수 | 약 30시간 |
| 중간값 | 약 114시간 (4.75일) |
| 75분위수 | 약 240시간 (10일) |
| 평균 | 약 146시간 (6.1일) |
| 최댓값 | 768시간 (32일) |

평균(146h)이 중간값(114h)보다 높은 것은 분포가 오른쪽으로 치우쳐 있음을 의미한다. 모델 학습에서는 `log1p` 변환 후 표준화하여 사용한다.

**수집 간격에 따른 측정 오차:**
스냅샷 수집이 6시간 간격이므로 `trending_duration_h`에는 구조적 오차가 존재한다. 
<br>실제 트렌딩 진입과 이탈 시각 모두 최대 6시간의 오차가 있어 총 오차는 최대 12시간이다. 회귀 모델의 RMSE 해석 시 반드시 고려해야 한다.

### 3-2. 결측값 분석

| 컬럼명 | 처리 방법 | 근거 |
|--------|----------|------|
| `title`, `channel_title` | 빈 문자열(`''`)로 대체 | 집계 과정 오류 방지 |
| `comment_count` | 0으로 대체 + `comment_count_missing` 플래그 생성 | YouTube 댓글 비공개 설정 시 null 발생. 0 대체 후 engagement_ratio 계산 논리 유지 |
| `view_growth_24h` | 24h 미관측 이벤트는 결측 유지 | T0-only 모델에서 미사용. 강제 대체 시 데이터 누수 |
| `video_id`, `channel_id`, `collection_date`, `published_at`, `view_count`, `rank` | 해당 행 전체 제거 | 이벤트 식별 또는 집계 불가 |

### 3-3. 이상값 분석

| 피처 | 탐지 방법 | 처리 방법 | 근거 |
|------|----------|----------|------|
| `view_count` | 조건 기반 | `view_count ≤ 0` 행 제거 | 음수·0 조회수는 데이터 오류 |
| `rank` | 조건 기반 | 1~200 범위 외 행 제거 | 트렌딩 순위 정의 범위 이탈 |
| `comment_count` | 조건 기반 | 음수 행 제거 | 데이터 오류 |
| `latency_to_trend_h` | 조건 기반 | 0 미만 값 제거 | 업로드 시각이 트렌딩 진입보다 늦은 것은 오류 |
| `trending_duration_h` | 조건 기반 | 6시간 미만 제거 | 스냅샷 수집 간격(6h) 미만은 수집 오류로 판단 |
| `pretrend_view_velocity` | 분모 clip | latency를 최소 1h로 clip | latency가 0에 가까울 때 나눗셈 폭발 방지 |
| EDA용 수치 피처 | Winsorization | 1~99% 범위 클리핑 | EDA 시각화·통계 요약 왜곡 방지 전용. **모델 학습 입력에는 미적용** |

**이상값 처리 원칙:** 모델 학습용 피처와 EDA 전용 변수를 명확히 구분한다. Winsorization을 적용한 변수는 `_wins_eda` 접미사를 붙여 모델 입력 변수와 분리한다.

### 3-4. 카테고리 그룹별 특성

| category_group | 포함 원본 카테고리 | 트렌딩 지속 특성 | 소비 패턴 |
|----------------|-----------------|----------------|----------|
| Entertainment | Entertainment(24), Sports(17), Gaming(20), Film&Animation(1), Comedy(23) | 중간 (편차 큼) | 트렌드·밈 의존. 외부 SNS 유입 시 지속 연장 가능 |
| Music | Music(10) | 짧은 편 | 팬덤 기반 초기 폭발 후 급락. 커버·반응 영상의 동시 진입으로 포화도 상승 |
| Lifestyle | Travel(19), Pets(15), Autos(2), People&Blogs(22), Howto&Style(26) | 중간~긴 편 | 정보 탐색형. 검색 유입 지속 |
| Education | Education(27), Science&Technology(28) | 긴 편 | 저장 후 재방문 구조. 알고리즘의 지속 추천 |
| News | News&Politics(25), Nonprofits&Activism(29) | 짧음 | 이슈 소멸 시 급락. 진입 속도는 가장 빠름 |


### 3-5. 상관관계 분석

**trending_duration_h와 주요 피처 간 예상 Spearman 상관**

| 피처 | 예상 방향 | 근거 |
|------|----------|------|
| `view_growth_24h_log` | 양(+) | 초기 반응 지속 시 알고리즘 추가 노출 → 지속 연장 |
| `latency_to_trend_log` | 양(+) | 오래 걸려 진입한 영상은 검색 유입형으로 지속성 높음 |
| `pretrend_view_velocity_log` | 양(+) | 트렌딩 진입 전 조회 속도가 빠를수록 지속성 높음 |
| `saturation_index_30d_mean_prev` | 음(-) | 포화도 높을수록 경쟁 심화 → 지속 단축 |
| `T0_engagement_ratio_log` | 양(+) | 초기 커뮤니티 반응이 클수록 지속 경향 |

`trending_duration_h`가 정규분포가 아닌 왜곡 분포를 따르므로 순위 기반 Spearman 계수를 사용한다.

**다중공선성 점검 대상:**
`T0_view_log`와 `pretrend_view_velocity_log`는 T0_view가 분자에 포함되어 상관될 수 있다. VIF 확인 후 판단한다.

### 3-6. EDA 주요 발견사항 (Insight)

- **인사이트 1 (RQ1 근거):** 카테고리는 트렌딩 지속 시간의 구조적 결정 인자다. Education과 Music 간 지속 시간 분포 차이가 통계적으로 유의미할 것으로 예상된다.
- **인사이트 2 (RQ2·3 근거):** T0 시점 조회수와 카테고리만으로도 일정 수준의 예측이 가능하다. 진입 후 24시간 조회수 성장을 추가하면 예측력이 개선되는지가 모델 비교의 핵심이다.
- **인사이트 3:** 포화도(`saturation_index_30d_mean_prev`)가 높은 날 진입한 이벤트는 평균 지속 시간이 짧은 경향이 있다. Music 카테고리에서 이 효과가 가장 강하게 나타날 것으로 예상된다.
- **인사이트 4:** `latency_to_trend_h`가 긴 이벤트는 검색 유입형 콘텐츠일 가능성이 높으며, 지속 시간이 상대적으로 길다. Education에서 이 패턴이 두드러질 것으로 예상된다.
- **인사이트 5:** 업로드 시간대(KST 기준) 저녁 7~10시 구간에서 트렌딩 진입 이벤트가 집중된다. 이 시간대 진입 영상의 초기 반응 속도가 가장 빠른 경향이 있다.

## 4. 데이터 전처리

### 4-1. 전처리 설계 원칙

**원칙 1 — 원본 불변:**
원본 스냅샷 테이블(`trending_snapshots`)은 절대 수정하지 않으며 read-only로 유지한다. 모든 파생 변수는 별도 테이블(`video_trending_events`)에 저장한다.

**원칙 2 — 데이터 누수 차단:**
예측 모델에 투입되는 모든 X변수는 반드시 트렌딩 진입 시점(T0) 또는 그 이전에 알 수 있는 값만 사용한다. T0 이후에야 알 수 있는 변수는 분석용 테이블에는 포함하되 모델 입력에서 전면 제외한다.

**원칙 3 — T0-only / T0+24h 모델 분리:**
`view_growth_24h_log`는 T0+24시간이 경과한 후에야 알 수 있다. 이 변수를 전체 이벤트에 강제로 채우거나 T0-only 모델에 포함하면 데이터 누수다. 두 모델을 완전히 분리하여 독립적으로 학습한다.

### 4-2. STEP 1 — 기본 정제

| 처리 항목 | 처리 방법 | 근거 |
|----------|----------|------|
| 시간 타입 변환 | `collection_date`, `published_at` UTC → KST(Asia/Seoul) 변환 후 timezone 제거 | KR 트렌딩 분석이므로 한국 시간 기준으로 통일 |
| 수치형 변환 | `view_count`, `comment_count`, `rank`, `category_id` → `pd.to_numeric(errors='coerce')` | 문자열 혼입 방지 |
| 필수값 결측 제거 | `video_id`, `channel_id`, `collection_date`, `published_at`, `category_id`, `view_count`, `rank` 결측 행 제거 | 이벤트 식별 또는 집계 불가 |
| 비정상 수치 제거 | `view_count` ≤ 0, `rank` 범위 1~200 외 제거 | 정의 범위 이탈 및 오류 데이터 |
| 중복 스냅샷 제거 | 동일 (video_id, collection_date) 중복 시 rank가 낮은(좋은) 행 유지, rank가 같으면 view_count가 큰 행 유지 | 동일 시점 중복 수집분 정리 |
| `comment_count` 결측 처리 | 결측 → 0 대체, `comment_count_missing` 플래그 생성 | 댓글 비공개 설정 영상 처리 |
| `comment_count` 음수 제거 | 음수 행 제거 | 데이터 오류 |
| 불필요 컬럼 제거 | `description`, `default_audio_language`, `live_broadcast_content` 제거 | 분석 미사용 |

### 4-4. STEP 3 — 재진입 이벤트 분리

동일 영상이 트렌딩 차트에서 이탈 후 재진입하는 경우를 별도 이벤트로 분리한다.

| 처리 항목 | 내용 |
|----------|------|

| 처리 | 첫 스냅샷 또는 6시간 초과 공백 발생 시 `new_event = True`, `video_id` 내 cumsum으로 `event_id` 부여 |

트렌딩 진입 후 24시간 동안의 조회수 증가량을 계산한다. 이 변수는 T0+24시간이 경과한 후에야 알 수 있으므로 T0-only 모델에는 절대 사용하지 않는다.

|----------|------|

| 음수 처리 | `raw_view_growth_24h < 0` → 0으로 클리핑 |

| `has_24h_observation` 사용 제한 | **필터링 조건으로만 사용. 예측 피처로 투입 금지** (Y 정보 간접 포함) |

### 4-8. STEP 7 — 타겟 변수 생성

**Y1: trending_duration_h (회귀 타겟)**
T_end - T0의 실측값이다. 추정이나 proxy가 아닌 직접 측정값이므로 회귀 타겟으로 타당하다. 학습 시 `log1p` 변환 후 표준화하여 사용하며, 수집 간격 구조상 최대 12시간의 측정 오차가 존재한다.

**Y2: TDI — Trending Duration Index (복합 지수)**

단순 지속 시간만으로는 카테고리 간 비교가 불공정하다. Education은 구조적으로 오래 가고 Music은 구조적으로 짧기 때문이다. TDI는 이를 보정한 0~1 사이의 복합 지수다.

```
TDI = (trending_duration_h / cat_q95_duration) × (1 - (best_rank - 1) / 200)
```

- `cat_q95_duration`: 카테고리별 지속 시간 95분위수 → 카테고리 내 상대적 위치 환산
- `best_rank`: 이벤트 기간 내 최고 순위 (낮을수록 좋은 순위) → 순위 품질 보정
- 결과 0~1 범위로 clip

**TDI가 메인 타겟인 이유:**
TDI는 단순히 "오래 지속되는가"가 아니라 "지속성과 순위 품질을 모두 갖춘 성과형 트렌딩 이벤트인가"를 측정한다. 
<br>단순 장기 노출 영상이 아니라, 높은 순위를 유지하며 오래 지속된 영상이다. TDI는 이 두 조건을 동시에 반영하는 유일한 지표다.

TDI 계산에 `best_rank`(사후 정보)가 포함되므로, TDI 자체는 예측 Y변수로만 사용하고 예측 X피처로는 절대 사용하지 않는다.

**Y3: tdi_label (메인 분류 타겟)**
TDI ≥ 0.4 → 1(성과형), TDI < 0.4 → 0(비성과형)

임계값 0.4 선택 근거:

| 임계값 | 0 비율 | 1 비율 | 1그룹 최솟값 | 두 그룹 중간값 배율 |
|--------|--------|--------|------------|------------------|
| 0.3 | 54.8% | 45.2% | 약 78h | 약 7.2배 |
| **0.4** | **65.0%** | **35.0%** | **약 108h** | **약 5.9배** |
| 0.5 | 74.4% | 25.6% | 약 138h | 약 5.1배 |

- 0.3 미선택: 1그룹 비율이 45%로 두 그룹 경계가 불명확
- 0.5 미선택: 74:26 클래스 불균형. 분류 모델이 0을 과도 예측하는 방향으로 편향 위험
- **0.4 선택:** ① 65:35의 허용 가능한 클래스 비율 ② 1그룹 최솟값 108h이 전체 중간값 114h와 근접 ③ 두 그룹 중간값 간 약 6배의 명확한 분리

**Y4: long_label 시리즈 (보조 비교용 분류 타겟)**

| 변수명 | 기준 | 의미 |
|--------|------|------|
| `long_label_global_q75` | 전체 이벤트 기준 상위 25% | 전체 KR 트렌딩 시장에서 오래 간 영상인가 |
| `long_label_cat_q75` | 각 카테고리 내 상위 25% | 같은 카테고리 안에서 오래 간 영상인가 |
| `long_label_48h` | 48시간 이상 지속 | 고정 기준 비교용 |

RQ1(카테고리별 지속 시간 차이 검정)에는 라벨이 아닌 원래 연속값 `trending_duration_h`를 직접 사용한다.

### 4-9. STEP 8 — Saturation Index 계산

| 변수명 | 계산 방법 | 사용 가능 여부 |
|--------|----------|--------------|
| `saturation_index_same_day_eda` | 당일 카테고리 이벤트 수 / 당일 포함 rolling 30일 최대값 | ❌ EDA 전용. 당일 미래 정보 포함 |
| `saturation_index_prev` | 전날 카테고리 이벤트 수 / 전날까지 rolling 30일 최대값 | 가능하나 일별 변동성 큼 |
| **`saturation_index_30d_mean_prev`** | **전날까지 rolling 30일 평균 / 전날까지 rolling 30일 최대** | **✅ 최종 모델 사용. 더 안정적** |

### 4-10. 인코딩 및 스케일링

| 변수명 | 변환 방법 | 근거 |
|--------|----------|------|
| `category_group` | One-Hot Encoding (drop_first=True) | 명목형 변수. 기준 카테고리 제외로 다중공선성 방지 |

### 4-11. 클러스터링 시도 및 누수 이슈 (RQ4)

RQ4(카테고리 내 트렌딩 패턴 유형화)를 위해 K-Means 클러스터링을 수행하였다. 클러스터링 생성에 사용한 변수는 다음과 같다.

```
trending_duration_h, entry_rank, view_growth_24h,
engagement_ratio, latency_to_trend_h, saturation_index
```

도출된 `cluster_id`를 예측 모델의 X피처로 추가하는 방안을 검토했으나, **클러스터링 생성에 Y변수(`trending_duration_h`)와 T0 이후 사후 정보(`view_growth_24h`)가 포함되어 있어, 
<br>`cluster_id`를 예측 피처로 투입하면 타겟 정보가 간접 유입되는 데이터 누수가 발생**한다. 따라서 `cluster_id`는 예측 모델의 X피처에서 제외하고, **사후 EDA 및 모델 결과 해석용 그룹으로만 활용**한다.

## 5. 데이터셋 분리

### 5-1. 분리 전략 — T0-only 모델과 T0+24h 모델

| 구분 | 사용 이벤트 | 피처 수 | 핵심 차이 |
|------|-----------|--------|----------|
| T0-only 모델 | `has_24h_observation == True` 공통 subset | 7개 | `view_growth_24h_log` 미포함 |
| T0+24h 모델 | `has_24h_observation == True` 공통 subset | 8개 | `view_growth_24h_log` 추가 |

**공통 subset 사용 이유:**
T0-only vs T0+24h 모델을 공정하게 비교하기 위해 두 모델 모두 동일한 이벤트 집합에서 학습·평가한다. 
<br>따라서 이때의 T0-only 모델은 전체 이벤트 기준이 아니라, **24h 관측 가능한 이벤트 subset 내에서의 T0-only 기준선**이다.

**선택 편향 주의:** 이 subset은 24시간 이상 트렌딩에 머문 이벤트들이므로 애초에 "어느 정도 지속된 영상들"이다. 결과 해석 시 이 한계를 명시한다.

### 5-2. Train / Test 분리

| 항목 | 내용 |
|------|------|
| 분리 비율 | Train 80% / Test 20% |
| Stratify 기준 | `category_group` (카테고리별 샘플 비율 유지) |
| Random Seed | 42 |

### 5-3. 최종 입력 피처 목록

**T0-only 모델 피처 (9개)**

| 피처명 | 타입 | 설명 |
|--------|------|------|
| `category_group` | 범주형 → OHE | 카테고리 그룹 (5개 → 4개 더미) |
| `entry_rank_log` | 수치형 | 진입 시점 조회수 (log 변환) |
| `T0_view_log` | 수치형 | 진입 시점 댓글수 (log 변환) |
| `T0_engagement_ratio_log` | 수치형 | 진입 시점 댓글/조회수 비율 ; 참여율 (log 변환) |
| `latency_to_trend_log` | 수치형 | 업로드→진입 시간 (log 변환) |
| `pretrend_view_velocity_log` | 수치형 | 진입 전 조회 축적 속도 (log 변환) |
| `weekday_sin / weekday_cos` | 주기 인코딩 | 요일 주기적 특성, sin/cos변환을 통한 7일 주기 연속성 보존 |
| `hour_sin / hour_cos` | 주기 인코딩 | 시간 주기적 특성, sin/cos 변환을 통한 24시 주기 연속성 보존 |
| `saturation_index_30d_mean_prev` | 수치형 (0~1) | 전날까지 30일 평균 카테고리 포화도 |

**T0+24h 모델 피처 (10개)**

T0-only 모델 피처 전체 + 아래 1개 추가:

| 피처명 | 타입 | 설명 |
|--------|------|------|
| `view_growth_24h_log` | 수치형 | 진입 후 24시간 조회수 증가량 (log 변환) |

### 6-2. 전처리 파이프라인 요약

```
[원본 스냅샷: trending_snapshots.parquet]  ← 읽기 전용, 수정 금지
    │
    ├─ STEP 1: 기본 정제 (UTC→KST, 결측·이상·중복 제거)
    ├─ STEP 2: 카테고리 정규화 (category_id → 5그룹 category_group)
    ├─ STEP 3: 재진입 이벤트 분리 (6h 초과 공백 → 별도 event_id)
    ├─ STEP 4: view_growth_24h 생성 (has_24h_observation == True만)
    ├─ STEP 5: 이벤트 단위 집계 (groupby video_id + event_id)
    │          → T0, T_end, entry_rank, T0_view, T0_comment 집계
    │          → trending_duration_h (Y1) 계산
    │          → T0_engagement_ratio, latency_to_trend_h 등 파생 피처
    ├─ STEP 6: 추가 파생 피처 (pretrend_view_velocity_log 등)
    ├─ STEP 7: 타겟 변수 생성
    │          → trending_duration_h (Y1, 회귀)
    │          → TDI → tdi_label (Y2, 메인 분류 타겟)
    │          → long_label 시리즈 (보조 비교용)
    ├─ STEP 8: Saturation Index 계산 및 JOIN
    │          → saturation_index_30d_mean_prev (예측용, 최종 모델 사용)
    │          → saturation_index_same_day_eda (EDA 전용)
    └─ STEP 9: 최종 저장 (분석용 / T0-only 모델용 / T0+24h 모델용 분리 / 원본 스냅샷 / 포화도)

[클러스터링 별도 수행] → cluster_id 생성
    → 누수 이슈 (Y변수·24h 사후 정보 포함) 확인
    → 예측 피처 제외, 사후 EDA 및 결과 해석용 그룹으로만 활용

[전처리 완료] → T0-only / T0+24h 모델 학습 (공통 subset 기준)
```
